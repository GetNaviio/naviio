## New / regressed issues since 2026-06-17
**None.** The one read-only audit agent scanned all 97 `src/app/api/**/route.ts` files plus the lib layer and found no regressions in any of the targeted classes:

- **(a) Unauthenticated routes doing DB/Anthropic work — none.** Every Prisma/Anthropic route is guarded by `requireAuth()` (`src/lib/auth.ts:182`) or a `withOrg`/`withAuth`/`withOwner` wrapper (`src/lib/api/with-org.ts`). All session-less DB routes are legitimately public with their own protection: pre-auth (`auth/login`, `auth/register`, webauthn flows), `auth/me` (returns user-or-null), `health`, cron routes (verify `Bearer ${CRON_SECRET}`), webhooks (`auth/stripe/webhook`, `credits/webhook`, `auth/plaid/webhook` — signature-verified), `portal/[token]` & `org/invites/preview` (token-hash, rate-limited), and OAuth callbacks (validate `state`). The new Navi routes (`/api/navi/decision`, `/api/navi/extract`) are guarded and credits-metered.
- **(b) Lowercase Prisma `integration` provider literals — none.** All `provider:` where/data literals use the UPPERCASE enum. Dynamic mapping is correct, incl. `ghl`/`gohighlevel` → `GOHIGHLEVEL` (`src/app/api/integrations/disconnect/route.ts:11-23`, `src/lib/integrations/refreshToken.ts:159`). Lowercase slugs in `integrations/page.tsx` are UI catalog ids, not query literals.
- **(c) user.id passed where orgId expected — none.** Every route resolves org via `getDefaultOrgId(user.id)` and passes the resulting cuid to org-scoped functions. No `orgId: user.id` swaps found.
- **(d) Money-unit (cents vs major) mixups — none.** Stripe cents are divided by 100 before joining the shared ledger / major-unit math (`src/lib/integrations/stripe-map.ts:25`, `stripe.ts:146,231-232,285`). The new "Sync Now persists charges to the ledger" path goes through `mapStripeCharge` (cents→major), so it is consistent.
- **(e) Newly hardcoded secrets in `src/` — none.** No live key/token/password literals. The only secret-like literal, `'dev-secret-change-in-production'` (`src/lib/auth.ts:20`), is correctly gated — `getJwtSecret()` throws in production and only returns the dev default in non-prod.
