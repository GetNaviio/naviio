## New / regressed issues since 2026-06-18
**None.** One read-only audit agent concentrated on the new billing/Stripe/Navi/benchmarks/categorization code (the highest-churn, highest-risk surface this run) and swept the lib layer + all 123 `src/app/api/**/route.ts` files. Findings by class:

- **(a) Unauthenticated routes doing DB/Anthropic work — none.** Every new money/auth-surface route is gated: billing `subscribe`/`confirm` use `withOwner`/`withAuth`, `billing` GET uses `withOrg`, the plan-billing `webhook` is Stripe-signature-verified (`constructPlanBillingEvent`). Firm billing/connect routes all use `withAuth`; both firm webhooks verify signatures (`constructFirmBillingEvent`, `constructConnectEvent`). Navi `agent`/`board-pack` use `requireAuth` + `getDefaultOrgId`; `action`/`decision` use `withOrg`; Anthropic is only reached inside the auth'd + credit-metered agent path. Benchmarks read route uses `requireAuth`; the benchmarks cron verifies `Bearer ${CRON_SECRET}` via `timingSafeEqual`. The only prisma-importing routes without an auth helper are webauthn pre-auth + `health` — both on the legitimate-public allowlist.
- **(b) Lowercase Prisma `integration` provider literals — none.** All `Integration` queries use the UPPERCASE enum. `provider: 'dropbox'` targets the separate `DocumentSource`/`DocumentRef` tables; `'ghl'` literals are token-store slugs; `'google'`/`'workos-sso'` are `Account.provider` (OAuth) — none are `Integration.provider` query literals.
- **(c) user.id passed where orgId expected — none.** The new multi-entity code is consistent: `countOwnedOrgs(user.id)`/`getOwnerBillingOrg(user.id)`/`getOwnerPlan(user.id)` correctly take the userId, while org-scoped calls (`getOrgBilling(orgId)`, `syncPlanSubscriptionQuantity(subId, …)`) take the org cuid / sub id. `org/switch` validates membership before using a supplied orgId.
- **(d) Money-unit (cents vs major) mixups — none.** *(Highest-risk class this run given all the new pricing code.)* `mapStripeCharge` divides `netCents / 100` before persisting to the ledger, and the ledger-persist path routes every charge through it. All Stripe MRR/revenue/refund math divides cents by 100 (`stripe.ts:146,231,289`). New plan/firm pricing stays in integer cents end-to-end (`lib/billing/plans.ts`, `lib/firm/billing.ts`): graduated overage, annual ×10, commission, and `planCostForEntities`/`computeFirmBill` never cross units; Stripe `quantity` values are entity/org *counts*, not amounts. Every display layer (`PlanSection.tsx`, `BillingSection.tsx`, the `fmtUSD` helpers) divides cents by 100. Benchmarks operate purely on major-unit ledger amounts.
- **(e) Newly hardcoded secrets in `src/` — none.** No live keys, price ids, webhook secrets, or tokens. Stripe price ids resolve from env (`STRIPE_PLAN_PRICE_*`, `STRIPE_FIRM_PRICE_*`); secret keys from `process.env`. The only `sk_test_`/`rk_test_` strings are prefix-validation literals in the env-gated dev component `TempStripeKeyConnect.tsx` (no actual key value). The gated `'dev-secret-change-in-production'` in `auth.ts` remains correctly env-gated.
